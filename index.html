Selected work

Self-directed crypto research, Feb–Jun 2026.

  • The phantom-fill trap

    Touch-based backtest engines fill you on any trade at your price. Exchanges don’t work that way.

    An Avellaneda–Stoikov market-maker backtested at Sharpe −3.5. I suspected the engine was filling on transient price-touches, rebuilt the fill model from order-book-delta-inferred trades, and showed ~65% of the fills were phantom. The strategy was never the problem — the simulation was lying.

    • hftbacktest
    • Python

  • 269 commits, and the answer was “no”

    I ran the test that could kill my own idea. It did.

    An ML-gated volatility-breakout strategy that looked clean on paper. A numba slippage model calibrated from 700+ real fills, plus a self-inflicted lookahead leak I caught, flipped the edge negative across all 42 parameter cells. I killed the strategy class and shipped a reproducible negative-result package.

  • The corruption detective

    Model won’t learn? Suspect the data before you touch the model.

    A model abstaining on every fold — so I audited the data, not the model. A streaming pass over 3 years of exchange archives found 33% non-monotonic timestamps and 13M+ duplicate trades inflating volatility features. I patched the panels and published both the contaminated and corrected results.

    • data-integrity
    • PyArrow
    • pandas

  • An AFML pipeline on HPC

    I wrote the leakage auditor before I wrote the strategy.

    CUSUM event sampling, triple-barrier labels, purged/embargoed CV, sample weights, and a non-negotiable lookahead auditor gating every stage — deployed to a production config on a SLURM cluster that held up under 16 bps round-trip cost stress.

    • LightGBM
    • SLURM
    • AFML

  • Memory-leak archaeology

    A collector running 311 WebSocket streams for days needs to not OOM. This one did.

    An unbounded leak in a 311-WebSocket async collector, traced past Python’s tracked allocator to PyArrow C++ column metadata in the glibc heap. Three fixes cut RSS drift from 2.4 GB to ~0 over a 104-hour run.

    • asyncio
    • systems
    • glibc

  • Limit-order-book matching engine

    Five C++ implementations. Only the ones that produce byte-identical trades to the reference get a speedup claim.

    Five layered C++17 implementations of a price-time-priority LOB (reference → slow → broken → optimised → lock-free concurrent reads) sharing a single deterministic workload. A full-trade-stream checksum confirms bit-identical execution across correct engines and catches the intentional mis-pricing bug. The optimised engine uses a flat price ladder, intrusive node pool, and O(1) cancel; CONSTRAINTS.md documents exactly where its assumptions break down.

    • C++17
    • matching-engine
    • price-time priority
    • seqlock
